# Git Flashcards — Карточки для запоминания (Дни 5-8)

## Карточка 1: Три уровня Git
**Вопрос:** Назовите три уровня Git и их назначение.

**Ответ:**
1. **Working Directory** — где вы работаете (ваши файлы)
2. **Staging Area** — где файлы готовятся к сохранению (`git add`)
3. **Repository** — где хранится история (`.git` папка)

---

## Карточка 2: git add vs git commit
**Вопрос:** В чём разница между `git add` и `git commit`?

**Ответ:**
- **git add** — добавляет файлы в staging area (подготовка)
- **git commit** — сохраняет файлы в репозиторий (финализация)

---

## Карточка 3: git status вывод
**Вопрос:** Что означают красные и зелёные файлы в `git status`?

**Ответ:**
- **Красные** — файлы не добавлены в staging (untracked/modified)
- **Зелёные** — файлы добавлены в staging (staged)

---

## Карточка 4: Создание ветки
**Вопрос:** Как создать новую ветку и сразу на неё переключиться?

**Ответ:**
```bash
git switch -c feature/name
# или старый способ:
git checkout -b feature/name
```

---

## Карточка 5: Слияние веток
**Вопрос:** Как слить ветку `feature/day3` в `main`?

**Ответ:**
```bash
git switch main              # переключитесь на main
git merge feature/day3       # слейте feature/day3 в main
```

---

## Карточка 6: Fast-forward merge
**Вопрос:** Что такое Fast-forward merge?

**Ответ:**
Это когда Git просто переместит указатель main на вашу ветку, потому что между ними нет конфликтующих изменений. Это быстрое и чистое слияние.

---

## Карточка 7: Удаление ветки
**Вопрос:** Как удалить ветку `feature/old-feature`?

**Ответ:**
```bash
git branch -d feature/old-feature      # нормальное удаление
git branch -D feature/old-feature      # принудительное удаление
```

---

## Карточка 8: Remote репозиторий
**Вопрос:** Что такое `origin` в Git?

**Ответ:**
`origin` — это имя удалённого репозитория (обычно на GitHub). Это просто сокращение для полного URL вашего репозитория.

---

## Карточка 9: git push впервые
**Вопрос:** Как загрузить новую ветку на GitHub впервые?

**Ответ:**
```bash
git push -u origin feature/name
# -u флаг устанавливает отслеживание (upstream)
```

---

## Карточка 10: git pull vs git fetch
**Вопрос:** В чём разница между `git pull` и `git fetch`?

**Ответ:**
- **git fetch** — только получает информацию с GitHub (не меняет ваши файлы)
- **git pull** — получает И слевает изменения в вашу ветку (fetch + merge)

---

## Карточка 11: Отмена коммита
**Вопрос:** Как отменить последний коммит но сохранить файлы?

**Ответ:**
```bash
git reset HEAD~1
# Ваши файлы останутся, но коммит будет отменён
```

---

## Карточка 12: Отмена staged файла
**Вопрос:** Я добавил файл в staging (`git add`). Как его оттуда убрать?

**Ответ:**
```bash
git restore --staged <файл>
# или старый способ:
git reset <файл>
```

---

## Карточка 13: .gitignore
**Вопрос:** Что такое .gitignore и зачем он нужен?

**Ответ:**
`.gitignore` — это файл, где указаны файлы/папки которые Git должен игнорировать (не отслеживать). Примеры: `node_modules/`, `.env`, `*.log`

---

## Карточка 14: Коммит сообщения
**Вопрос:** Напишите 3 хороших сообщения коммита.

**Ответ:**
1. `Add arrow functions exercises from Day 1`
2. `Fix business card formatting issue`
3. `Refactor array methods with examples`

(Плохие: "update", "fix", "asdf")

---

## Карточка 15: git log
**Вопрос:** Как посмотреть историю коммитов в компактном виде?

**Ответ:**
```bash
git log --oneline
# Вывод:
# abc1234 Add Day 1 exercises
# def5678 Initialize repository
```

---

## Карточка 16: Ветки и их типы
**Вопрос:** Назовите 3 распространённых типа веток и их назначение.

**Ответ:**
1. **main** — основная ветка (готовый код)
2. **feature/name** — новые функции
3. **fix/name** или **bugfix/name** — исправления ошибок

---

## Карточка 17: git status перед push
**Вопрос:** Перед `git push` вы видите: "Your branch is ahead of 'origin/main' by 2 commits". Что это значит?

**Ответ:**
Это значит, что у вас есть 2 локальных коммита которые ещё не загружены на GitHub. Нужно сделать `git push`.

---

## Карточка 18: Конфликт слияния
**Вопрос:** Что произойдёт если две ветки изменили один и тот же файл и вы пытаетесь их слить?

**Ответ:**
Git создаст **конфликт слияния**. Вам нужно вручную выбрать какие изменения оставить, отредактировать файл, затем сделать `git add` и `git commit`.

---

## Карточка 19: SSH vs HTTPS
**Вопрос:** Какой способ подключения к GitHub безопаснее: SSH или HTTPS?

**Ответ:**
**SSH** безопаснее. Он использует пару ключей вместо пароля. Рекомендуется для регулярной работы.

---

## Карточка 20: GitHub Pull Request
**Вопрос:** Что такое Pull Request?

**Ответ:**
Pull Request (PR) — это способ предложить изменения. Вы создаёте PR из вашей ветки в main, коллеги могут посмотреть код, потом PR сливается в main.
